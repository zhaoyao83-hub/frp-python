# 从零构建轻量级反向代理：MyFRP 的技术演进之路

## 一、背景与目标

在当今分布式系统和远程办公的时代，内网穿透工具已成为开发者和运维人员的必备利器。FRP（Fast Reverse Proxy）作为业界知名的开源方案，以其高性能和丰富功能被广泛使用。然而，对于轻量级场景（如个人开发、小型团队协作），FRP 的 C++ 实现显得过于臃肿，且缺乏可视化管理能力。

基于此，我们启动了 **MyFRP** 项目——一个基于 Python asyncio 从零实现的轻量级反向代理工具。项目的核心理念是：**以 Python 的简洁性实现 FRP 的核心能力，并通过现代化的 Web 界面提供友好的管理体验**。

### 已完成的核心能力

通过前期开发，MyFRP 已实现以下核心功能：

| 功能模块 | 状态 | 关键技术 |
|---------|------|---------|
| TCP 端口转发 | ✅ 完成 | asyncio 异步读写 |
| UDP 端口转发 | ✅ 完成 | 连接抽象 + 数据报转发 |
| Token 认证 | ✅ 完成 | LOGIN/LOGIN_RESP 握手 |
| TLS 加密 | ✅ 完成 | 标准库 ssl 模块 |
| 断线重连 | ✅ 完成 | 指数退避算法 |
| 心跳保活 | ✅ 完成 | PING/PONG 定时检测 |
| 流量统计 | ✅ 完成 | 内存计数器 |
| 访问控制 | ✅ 完成 | 端口/IP 黑白名单 |

核心代码位于 [frps.py](file:///Users/zhaoyao/myfrp/frps.py) 和 [frpc.py](file:///Users/zhaoyao/myfrp/frpc.py)，仅依赖 Python 标准库，零第三方依赖即可运行。

## 二、方法论解读

### 2.1 启发式提示词驱动的开发

在项目初期，我们使用启发式提示词来引导技术方向：

```
"实现一个类似 frp 的反向代理内核"
```

这个提示词拆解为以下关键技术点：

1. **架构模式**：客户端-服务端主从架构
2. **通信方式**：控制通道 + 数据通道
3. **协议设计**：自定义消息协议
4. **并发模型**：高并发连接处理

通过对 FRP 核心原理的深入分析，我们选择了 **Python asyncio** 作为核心技术栈，理由如下：

- **单线程高并发**：asyncio 的事件循环机制适合处理大量 IO 密集型连接
- **代码简洁**：Python 的语法特性使网络编程代码更易读、易维护
- **生态丰富**：丰富的第三方库支持快速构建 Web 管理界面
- **学习曲线平缓**：相比 Go/C++，Python 的异步编程门槛更低

### 2.4 分布迭代：渐进式打磨路径

我们采用**分布迭代**的开发策略，将项目划分为四个递进的打磨阶段（P0-P3），每个阶段都有明确的目标、关键工作和预期工作量：

| 阶段 | 目标 | 关键工作 | 预计工作量 | 当前状态 |
|------|------|----------|-----------|----------|
| **P0 可用** | 能稳定跑通基础场景 | 断线重连、超时清理、Token 认证、错误处理 | 2-3 天 | ✅ 已完成 |
| **P1 生产级** | 可安全上线 | TLS 加密、访问控制、监控统计、优雅关闭 | 5-7 天 | ✅ 已完成 |
| **P2 高性能** | 可支撑高并发 | 二进制协议、数据通道分离、零拷贝、多进程 | 1-2 周 | 📋 规划中 |
| **P3 功能完善** | 媲美 frp | HTTP 代理、STCP/XTCP、Web 管理 | 3-4 周 | ⏳ 进行中 |

#### 分布迭代的核心原则

**1. 价值递增，风险递减**

每个阶段都是在前一阶段的基础上增量迭代，确保每一步都能交付可用价值：

```
P0 可用 → P1 生产级 → P2 高性能 → P3 功能完善
    │          │            │            │
    └──────────┼────────────┼────────────┘
               ▼            ▼
          稳定性提升    性能飞跃
               │            │
               └────────────┘
                    ▼
               功能完备
```

**2. 阶段内聚焦，阶段间衔接**

每个阶段聚焦于特定的核心目标，避免范围蔓延：

- **P0**：最小可用产品，核心代理功能能跑通
- **P1**：安全与可靠性，生产环境可用
- **P2**：性能优化，支撑大规模并发
- **P3**：功能完善，覆盖 frp 主流场景

**3. 迭代反馈，持续优化**

每个阶段完成后进行复盘，基于实际使用反馈调整下一阶段的优先级：

```
阶段开发 → 测试验证 → 用户反馈 → 复盘调整 → 下一阶段
    │           │          │          │
    └───────────┴──────────┴──────────┘
              闭环迭代
```

#### 当前迭代位置与下一步

**当前位置**：P1 生产级已完成，正在进入 P3 功能完善阶段（Web 管理面板）

**跳过 P2 的原因**：
- P2 的高性能优化需要基于实际生产负载数据，过早优化可能导致过度设计
- 当前用户规模较小，P1 的性能已足够支撑
- Web 管理面板的需求更迫切，能显著提升用户体验

**P3 阶段内部迭代**：

```
P3.1 核心可用 → P3.2 完善体验 → P3.3 功能完备
    │               │               │
    ├── 服务启停    ├── 连接监控    ├── HTTP 代理
    ├── 配置管理    ├── 代理监控    ├── STCP/XTCP
    ├── 鉴权系统    ├── 用户管理    └── 高级功能
    └── 日志监控    └── 流量趋势
```

这种分布迭代策略确保了项目始终处于**可用、可测、可交付**的状态，同时保持了灵活调整的能力。
