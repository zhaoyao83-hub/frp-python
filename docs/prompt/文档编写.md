### D005. 内网穿透原理：服务端无法主动连接客户端

**原始提示词**：
> ─── NEW_CONN(msg) ─ 服务端无法主动连接客户端（内网）； 如何解决

**提示词摘要**：
点出了内网穿透的核心矛盾——服务端在公网、客户端在内网，服务端无法主动连接客户端，问怎么解决。

**产出**（深度技术解释）：
- 核心原理：**反向连接 + 复用控制通道**
- 客户端主动发起连接（可穿透 NAT），服务端通过已有连接回传数据
- NAT 路由表的状态跟踪机制解释
- TCP 连接的双向性说明
- 为什么服务端不能直接连接客户端（NAT 阻止）
- 当前实现的优势：无需额外端口、自动穿透 NAT、简化部署、安全性高

**后续沉淀**：相关内容被整合进多篇文档的原理说明部分

### D006. 读写并发模型：多线程还是单线程 asyncio

**原始提示词**：
> 读写是两个不同线程吗，还是同一个线程用loop里 read write的形式

**提示词摘要**：
询问并发模型——读写操作是多线程实现还是单线程事件循环实现。

**产出**（深度技术解释）：
- 明确了是**单线程 + asyncio 事件循环**，读写是同一个线程内的不同协程
- 通过 `await` 让出执行权实现协作式并发
- 列出了服务端和客户端所有关键协程及其读写操作
- 对比了 asyncio vs 多线程的优劣
- 解释了为什么不用多线程：GIL 限制、切换开销、共享状态需要锁

**设计启示**：
- 这个问题揭示了 MyFRP 并发模型的核心设计选择
- 对于理解系统性能瓶颈和扩展方向很重要

## 第四阶段：方案探索（2026-07-11）

### D012. 远程命令行映射到 Web 管理

**原始提示词**：
> 要把远程的命令行 映射到web管理下； 该如何实现， 有哪些技术方向； 先讨论方案，不直接执行

**提示词摘要**：
探讨如何将远程机器的命令行终端集成到 Web 管理面板中，先做技术方案讨论，不写代码。

**产出**（方案讨论）：
- **4 个技术方向**的对比分析：
  1. **xterm.js + WebSocket + PTY**（自研/半自研）：最灵活，开发量大
  2. **ttyd / gotty**（现成工具 + FRP 端口映射）：零开发，开箱即用
  3. **基于 SSH 的 Web 封装**（Gateway 模式）：安全性好，堡垒机模式
  4. **Apache Guacamole**（重型方案）：功能最全，太重
- **3 种架构模式**：终端能力在远程机器 / 终端能力在服务端 / 纯端口映射
- **建议三步走**：先用 ttyd+FRP 验证需求 → 服务端加 Web 终端管理页 → 运维级能力考虑 SSH 网关
- **关键设计点**：协议扩展（SHELL 类型）、客户端 PTY、服务端 WS 网关、前端 xterm.js

**设计特点**：
- 先讨论方案再动手，避免方向走偏
- 多方向横向对比，给出选型建议
- 渐进式落地路线图

## 文档编写模式总结

### 模式对比

| 模式 | 说明 | 对应提示词 |
|------|------|-----------|
| **一次性出文档** | 给定范围，直接生成完整文档 | D001、D009、D010、D011 |
| **问答驱动文档化** | 先问答深入理解，再沉淀为文档 | D002~D006 → connection_flow.md、new_conn_detail.md |
| **问题→方案文档** | 提出一个问题/需求，输出完整方案设计 | D007（数据通道分离）、D012（Web 终端） |
| **元文档** | 复盘文档本身 | D013（本文档） |

### 文档产出清单

| 文档 | 类型 | 规模 | 触发方式 |
|------|------|------|----------|
| README.md | 项目说明 | 中 | 一次性出文档 |
| design.md | 详细设计 | 大 | 一次性出文档 |
| connection_flow.md | 流程说明 | 中 | 问答驱动 |
| new_conn_detail.md | 深度解析 | 中 | 问答驱动 |
| extend_data_channel.md | 扩展方案 | 大 | 问题→方案 |
| p0_production_ready.md | 阶段设计 | 大 | 一次性出文档 |
| p1_enterprise.md | 阶段设计 | 大 | 一次性出文档 |
| p2_high_performance.md | 阶段设计 | 大 | 一次性出文档 |
| 文档编写.md | 复盘文档 | 中 | 元文档 |

### 编写效率与质量观察

1. **给定清单式的提示词效率最高**：P0/P1/P2 那种"级别 + 能力清单"的格式，AI 能快速输出结构化的设计文档
2. **问答式能挖到更深的细节**：NEW_CONN 那种从一句话问到 12 个子阶段，深度远超一次性文档
3. **方案讨论类先讲方向再动手**：可以避免走偏，比如 Web 终端的四方向对比
4. **文档和代码需要双向校准**：写完代码后，文档很容易过时，需要定期做一致性检查（P005 文档检查）
