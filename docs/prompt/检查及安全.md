### C004. 代理规则不生效排查：新增 TCP 映射为什么没生效

**提示词原文**：
> 添加了一条 TCP 端口映射规则，名称为 web，本地地址 127.0.0.1:8086，远程端口 9096，但没有生效

**提示词摘要**：
> 排查用户在 Web 面板新增的 TCP 映射规则不生效的问题，定位根因并给出解决方案。

**排查过程**：
1. 检查 frpc.json 配置是否正确写入 → 配置正确
2. 检查 frpc 进程是否运行 → 进程在运行
3. 检查 frps 的 allow_ports 配置 → 发现问题！

**根因**：
- frps.json 中 `allow_ports` 配置为 `7000-9000`
- 用户配置的远程端口 `9096` 超出了允许范围
- frps 拒绝了该端口的代理注册请求

**解决方案**（二选一）：
1. 修改远程端口到 7000-9000 范围内
2. 扩大 frps.json 中 `allow_ports` 的范围

**产出**：
- 定位到 allow_ports 端口范围限制问题
- 后续在前端 UI 中增加了端口范围提示（待优化项）

### C005. 服务状态不一致排查：显示未启动但映射通路仍然通畅

**提示词原文**：
> 现在服务端单条映射规则处于未启用状态 服务处于未启动状态， 映射通路仍然是通的， 可以访问响应服务端口且能转发数据

**提示词摘要**：
> 排查两个问题：(1) 为什么服务显示未启动但实际在转发数据；(2) 为什么禁用了映射规则但仍然生效。

**排查过程**：
```
lsof -i:7000 -i:7500 -i:7501 -i:9096
  → 发现 PID 27527 (frps) 和 PID 27763 (frpc) 在运行
  → 这些进程不是 Dashboard 的 ServiceManager 启动的
```

**根因分析**：

**问题1：服务显示未启动但实际在运行**
- 存在历史遗留进程（之前通过命令行启动的 frps/frpc）
- Dashboard 的 ServiceManager 只管理自己启动的进程
- 遗留进程仍在监听端口、转发数据，但 Dashboard 显示"未启动"

**问题2：禁用代理规则后仍然生效**
- 禁用操作只是修改了 frpc.json 配置文件
- 运行中的 frpc 进程不会自动热加载配置
- 需要重启 frpc 才能使新配置生效

**修复方案**：

**后端修复（service_manager.py）**：
- 新增 `_find_external_processes()` 方法：通过 ps 命令匹配"脚本名 + 配置名"检测遗留进程
- `status()` 新增 `has_external_process` 和 `external_pids` 字段
- `start()` 启动前检测遗留进程，存在则阻止启动（提示 PID）
- `start(kill_existing=True)` 支持先清理遗留进程再启动
- 新增 `kill_external()` 方法：仅清理遗留进程
- 新增 API：`POST /api/service/{name}/kill-external`

**前端修复**：
- `StatusCard.tsx`：新增「存在遗留进程」黄色警告状态
  - 显示遗留进程 PID 列表
  - 提供「清理遗留进程」和「清理并启动」两个操作按钮
- `PortMapping.tsx`：新增「立即重启 frpc」快捷按钮
  - 检测 frpc 运行状态
  - 配置变更后可一键重启生效

**验证结果**：
- ✅ 遗留进程检测准确
- ✅ 启动拦截正常工作
- ✅ 强制清理启动功能正常
- ✅ 前端 TypeScript 编译通过，Vite 构建成功

### S001. 公网部署安全评估：放在公网有哪些安全隐患需要防范

**提示词原文**：
> 是否有安全隐患；放在公网 有哪些需要防范

**提示词摘要**：
> 全面评估 MyFRP 部署在公网环境下的安全风险，列出所有需要防范的点，并给出加固方案。

**风险识别与加固方案**：

| 风险等级 | 风险点 | 描述 | 加固方案 |
|----------|--------|------|----------|
| 🔴 高危 | 控制端口暴露 | 7000 端口直接暴露在公网，可能被暴力破解 | 1. 强 token（至少 32 位随机字符串）<br>2. IP 白名单限制<br>3. TLS 加密传输 |
| 🔴 高危 | WebAPI 无认证 | 7501 端口 /stats 接口无鉴权，可直接获取代理信息 | 绑定 127.0.0.1，仅允许本地访问 |
| 🟠 中危 | 代理端口滥用 | 攻击者可通过代理转发攻击内网服务 | 1. allow_ports 限制端口范围<br>2. 定期审计代理列表 |
| 🟠 中危 | 管理面板弱密码 | 默认 admin/admin123 弱密码 | 首次登录强制修改密码 + 强密码策略 |
| 🟡 低危 | 控制协议明文 | 未启用 TLS 时控制消息明文传输 | 默认建议启用 TLS |
| 🟡 低危 | 日志泄露敏感信息 | 日志中可能包含 token、密码等 | 敏感字段日志脱敏 |
| 🟡 低危 | 速率限制缺失 | 无连接速率限制，可能被 DDoS | 考虑添加连接数限制和速率限制 |

**即时修复（高危项）**：
- ✅ WebAPI 端口绑定地址改为 `127.0.0.1`
- ✅ 默认配置中建议强 token
- ✅ 文档中增加安全部署指南

**产出**：
- 完整的公网安全风险评估报告
- 高危项即时修复完成
- 安全加固路线图（中低危项按优先级逐步实施）

### S002. WebAPI 端口认证漏洞修复：7501 端口绑定 0.0.0.0 导致情报泄露

**提示词原文**：
> WebAPI 端口无认证 — frps.py:700-734 webapi_port: 7501 绑定 0.0.0.0，任何人可直接 curl http://你的IP:7501/stats 获取代理列表、流量统计、连接信息。这是情报泄露，攻击者可据此了解你的服务结构。 绑定127

**提示词摘要**：
> 修复 WebAPI 端口的高风险安全问题：将监听地址从 0.0.0.0 改为 127.0.0.1，防止公网直接访问 /stats 接口。

**风险分析**：
- **攻击场景**：攻击者扫描端口发现 7501，直接 GET /stats
- **泄露信息**：代理列表名称/类型/端口、流量统计、活跃连接数、服务运行时长
- **危害等级**：🔴 高危（情报泄露 → 辅助进一步攻击）

**修复内容**：

1. **frps.json 配置修改**：
```json
{
    "webapi_port": 7501,
    "webapi_addr": "127.0.0.1"   // 从 "0.0.0.0" 改为 "127.0.0.1"
}
```

2. **config.py 默认值调整**：
- `webapi_addr` 默认值从 `0.0.0.0` 改为 `127.0.0.1`
- 安全默认原则：开箱即安全

3. **文档同步更新**：
- README 安全部署章节增加说明
- dashboard_spec.md 中明确 WebAPI 仅本地访问

**验证结果**：
- ✅ 公网无法访问 7501 端口
- ✅ 本地 Dashboard 仍可正常调用 /stats（同机部署）
- ✅ 不影响管理面板功能

### S003. 文件管理模块安全设计：多根目录白名单 + 路径遍历防护

**提示词原文**：
> webadmin增加一个文件管理模块

**提示词摘要**：
> 在 Web 管理面板中增加文件管理功能，同时确保安全性，防止路径遍历、未授权访问等安全问题。

**安全设计要点**：

| 安全措施 | 实现方式 | 防护目标 |
|----------|----------|----------|
| 根目录白名单 | `file_manager_roots` 配置项，只有列出的目录可访问 | 防止访问系统敏感文件 |
| 路径遍历防护 | 规范化路径后检查是否仍在根目录内 | 防止 `../../../etc/passwd` 攻击 |
| 角色权限控制 | 仅 admin 角色可见可用，viewer 角色完全看不到 | 最小权限原则 |
| 文件类型限制 | 仅允许编辑文本文件，禁止二进制文件在线编辑 | 防止上传/执行恶意代码 |
| 文件大小限制 | 在线编辑文件大小上限（如 1MB） | 防止大文件导致的性能问题 |
| 操作日志记录 | 所有文件操作写入 Dashboard 日志 | 审计追溯 |

**后端安全实现（files.py）**：
- `_resolve_and_validate_path()`：路径解析 + 安全校验的核心函数
  1. 拼接根目录和相对路径
  2. `os.path.realpath()` 规范化（解析符号链接和 `..`）
  3. 检查规范化后的路径是否仍在根目录下
  4. 不在范围内则抛出 403 错误

**前端安全设计**：
- 侧边栏菜单根据角色动态渲染（viewer 看不到「文件管理」）
- 后端 API 也做了角色校验（双重保险）
- 文件编辑仅支持文本类型，二进制文件只显示信息不可编辑

**验证结果**：
- ✅ 路径遍历攻击被拦截（`../` 无法跳出根目录）
- ✅ viewer 角色无法访问文件管理（前后端双重校验）
- ✅ 符号链接不会导致越权（realpath 规范化）

### S004. JWT 认证机制安全设计

**提示词摘要**：
> Web 管理面板的登录认证系统采用 JWT 方案，确保安全性。

**安全设计要点**：

| 措施 | 实现 | 说明 |
|------|------|------|
| 密码存储 | bcrypt 哈希 | 加盐哈希，不可逆，防彩虹表 |
| Token 算法 | JWT (HS256) | 服务端签名，防篡改 |
| Token 有效期 | 默认 12 小时 | 过期自动失效 |
| 刷新机制 | Refresh Token | 可配置是否支持无感刷新 |
| 首次登录强制改密 | `must_change_password` 字段 | 默认 admin/admin123 首次登录必须修改 |
| 权限分级 | admin / viewer 两级 | 管理操作仅 admin，viewer 只读 |
| 依赖项 | passlib + python-jose | 成熟的密码学库，不自行实现 |

**后端实现（auth.py）**：
- `get_auth_manager()`：单例认证管理器
- `verify_password()`：bcrypt 密码校验
- `create_access_token()`：JWT 签发
- `require_admin`：FastAPI 依赖，admin 角色校验
- 持久化回调：用户变更自动写回 webadmin.json

**验证结果**：
- ✅ 密码不以明文存储
- ✅ Token 不可伪造（签名验证）
- ✅ 过期 Token 自动失效
- ✅ 未登录无法访问受保护接口

## 三、检查与安全的经验总结

### 3.1 检查工作方法论

| 方法 | 适用场景 | 效果 |
|------|----------|------|
| **并行检查 + 并行修复** | 文档一致性等大量重复检查工作 | 6 个 agent 并行 → 效率提升约 4x |
| **分级处理** | 风险评估、Bug 排查 | 高危先修，中低危排队，资源效率最大化 |
| **先定位后动手** | Bug 排查类问题 | 先用工具定位根因，再动手改代码，避免盲目试错 |
| **验证闭环** | 所有修复工作 | 改完必测，验证通过才算完成 |

### 3.2 安全设计原则

1. **默认安全（Secure by Default）**：开箱即用的配置就是安全的，WebAPI 默认绑 127.0.0.1
2. **最小权限原则**：文件管理仅 admin 可见，viewer 完全看不到
3. **纵深防御**：前端权限校验 + 后端权限校验，双重保险
4. **攻击面最小化**：能不暴露的端口就不暴露（如 WebAPI）
5. **输入不信任**：所有用户输入都要校验（路径遍历防护、配置字段校验）
6. **审计可追溯**：操作有日志，出问题能复盘

### 3.3 常见问题与避坑

| 问题类型 | 典型表现 | 避坑方法 |
|----------|----------|----------|
| **配置与运行脱节** | 改了配置但服务没重启，以为没生效 | 配置变更后明确提示需要重启，并提供快捷重启按钮 |
| **进程管理盲区** | 手动启动的进程管理面板不知道，状态显示错乱 | 启动前检测遗留进程，检测到则阻止启动并提示清理 |
| **端口范围忽略** | 随便选个端口，结果被 allow_ports 拦了 | 前端增加端口范围提示，创建时校验 |
| **文档不同步** | 功能实现了文档没更，新人看文档踩坑 | 每个阶段收尾做一次文档一致性检查 |
| **安全靠后想** | 功能做完了才想安全，发现架构性问题 | 设计阶段就考虑安全，安全是架构的一部分 |
